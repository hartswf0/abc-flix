Ghost Editing Heuristics

# Heuristics for the Arc Splice / Ghost Editing Concepts

These heuristics translate the uploaded **QQQQ Nexus Stepping Stone Collector** into design rules for the control surface: not as a normal browser video editor, but as a ritualized system of visible sources, lit slices, ghost media, magnetic rails, missing bodies, and EDL traces. 

---

# 1. Visible Source Ritual Heuristic

**Rule:** A source should not merely load. It should awaken.

The source lifecycle should feel like:

> dormant file → armed source → lit slices → dragged fragment → temporal apparition

Design implications:

* Use “armed,” “visible,” “lit,” and “ready” language instead of generic “uploaded.”
* Make source activation visually dramatic.
* Show source presence even before perfect decoding.
* Treat each A/B/C/D source as a distinct character, not just a file slot.

**Avoid:** bland upload states like `file selected` or `media imported`.

**Test:** Can a user tell when a source has become active without reading a manual?

---

# 2. Ghost Media Heuristic

**Rule:** Failed media should still have presence.

The phrase **“decode failed — source still armed”** should become a core design principle, not an error edge case.

If a video cannot decode, the system should still allow it to exist as:

* placeholder
* ghost source
* absent reel
* proxy waveform
* missing-media memory
* relinkable artifact

**Design move:** A failed source gets a spectral visual treatment instead of disappearing.

Example states:

| State     | Meaning                               |
| --------- | ------------------------------------- |
| `ARMED`   | Source has been accepted              |
| `VISIBLE` | Video can be previewed                |
| `GHOST`   | Source exists but cannot fully appear |
| `MISSING` | Source was known but is absent        |
| `RELINK`  | Source needs a body again             |

**Avoid:** treating decode failure as a dead end.

**Test:** Can the user still compose with a broken or missing source?

---

# 3. Ritual Grid Heuristic

**Rule:** The fixed 24-slice density is not a limitation until proven otherwise.

The grid should feel like a constraint that creates thought.

Do not rush to add infinite zoom, frame-level precision, or elastic timelines. The fixed grid gives the tool a recognizable tempo.

**Design implications:**

* Keep the default timeline visibly segmented.
* Let each slot feel like a beat, cell, card, or altar position.
* Make empty cells meaningful.
* Allow precision only as a temporary magnification mode.

**Avoid:** turning the rail into a conventional NLE timeline.

**Test:** Does the grid make users think differently, or does it merely frustrate them?

---

# 4. Slice-as-Offering Heuristic

**Rule:** A slice is not just a unit of footage. It is a chosen fragment.

When the user drags a lit slice, the interface should frame that action as selection, extraction, and offering.

The drag state should show:

> source · source time · destination time · layer · duration

Example:

> `A:04s → rail 08s · layer 2`

**Design implications:**

* Hovering over a slice should make it feel available.
* Dragging should make it feel extracted.
* Dropping should make it feel consecrated into sequence time.

**Avoid:** generic drag-and-drop behavior with no symbolic feedback.

**Test:** Does placing a clip feel like making an editorial claim?

---

# 5. Magnetic Rail Heuristic

**Rule:** The rail should behave like a force field, not a passive timeline.

The user should feel that clips are caught by the rail.

**Design implications:**

* Show snap glow before drop.
* Highlight valid landing cells.
* Let invalid drops resist or repel.
* Make overlap/layer priority visible.
* Make the rail feel gravitational.

**Avoid:** silent snapping that surprises the user.

**Test:** Before releasing the pointer, can the user predict exactly where the slice will land?

---

# 6. Empty Time Heuristic

**Rule:** Absence is an editable material.

The timeline should not treat gaps as nothing. Gaps should read as deliberate empty sequence time.

Possible labels:

* `EMPTY TIME`
* `BLACK`
* `ABSENCE`
* `UNSUMMONED`
* `NO ACTIVE CLIP`

**Design implications:**

* The playhead crossing an empty region should show a monitor state.
* EDL export may include gap events.
* Empty slots should remain visually legible.
* Silence/black should feel compositional, not accidental.

**Avoid:** collapsing the sequence duration to only occupied clips.

**Test:** Can the user intentionally design with what is missing?

---

# 7. EDL-as-Score Heuristic

**Rule:** Export is not the final artwork. It is a performable trace.

The JSON EDL should be treated like a score, spellbook, fossil, or set of temporal instructions.

**Design implications:**

* Preserve missing media references.
* Export gaps if they matter.
* Include ghost-source states.
* Include source identity, not just file paths.
* Make EDL readable by humans, not only machines.

Possible EDL event language:

```json
{
  "event": 7,
  "source": "B",
  "source_state": "ghost",
  "source_in": "00:00:04",
  "sequence_in": "00:00:12",
  "duration": 1,
  "layer": 2,
  "meaning": "placeholder still active"
}
```

**Avoid:** treating export only as technical workflow glue.

**Test:** Could someone reconstruct, reinterpret, or perform the sequence from the EDL alone?

---

# 8. Relink-as-Interpretation Heuristic

**Rule:** Relinking should not be framed only as repair. It can be reinterpretation.

When a project loads without media, the absence should become part of the experience.

**Design implications:**

* Show missing sources as named ghosts.
* Let users relink with different media.
* Preserve the old source identity as memory.
* Show which clips depend on each missing body.
* Allow “perform without relinking” using proxies.

**Avoid:** blocking the project until all media is restored.

**Test:** Can the same EDL become a different edit when relinked to different source bodies?

---

# 9. No-Final-Render Heuristic

**Rule:** Lack of final render can protect the system from becoming ordinary.

This tool may be stronger as a live composition instrument, temporal sketchbook, or performance console.

**Design implications:**

* Prioritize playback, mutation, and exportable structure.
* Treat the running interface as the artwork.
* Let sequences be performed, not only finalized.
* Avoid adding a render queue too early.

**Avoid:** assuming every video tool must produce a flattened file.

**Test:** Is the most meaningful output the rendered video, or the act/structure of splicing?

---

# 10. Console Mythology Heuristic

**Rule:** The interface language should sustain the fiction of an instrument.

The uploaded document identifies the surface as a “machine cult”: black field, neon commands, armed sources, warning states, live slices. Preserve that mythology.

**Use language like:**

* `ARM SOURCE`
* `SOURCE VISIBLE`
* `SPLICE PLACED`
* `DROP ON RAIL`
* `GHOST MEDIA`
* `RELINK BODY`
* `EDL EXPORTED`
* `SEQUENCE INVOKED`

**Avoid:**

* `Upload complete`
* `Add clip`
* `Error loading file`
* `Save successful`
* `Preview unavailable`

**Test:** Does the system sound like a distinctive instrument, or generic software?

---

# 11. Mode-as-Mood Heuristic

**Rule:** Modes should feel like mental states, not just system states.

The current mode set — edit, play, scrub, drag, roll, trim — can become a cognitive weather system.

| Mode  | Mood             |
| ----- | ---------------- |
| Edit  | potential        |
| Play  | invocation       |
| Scrub | searching        |
| Drag  | extraction       |
| Roll  | memory shift     |
| Trim  | wound adjustment |
| Ghost | absent presence  |

**Design implications:**

* Each mode should have a visual atmosphere.
* The playhead, rail, monitor, and source strips should respond to mode changes.
* The footer mode label should be reinforced spatially.

**Avoid:** relying only on a small text label in the HUD.

**Test:** Can the user feel the mode change before reading it?

---

# 12. Cut-as-Wound Heuristic

**Rule:** A cut should leave evidence.

Do not make clips feel like clean rectangles only. They should carry the memory of extraction.

**Design implications:**

* Angled clip edges are good; keep them.
* Trim handles should look like exposed scar tissue.
* Used source slices should be marked.
* Moving a clip should preserve its source wound: where it came from, what it took.

**Avoid:** overly polished card components that erase the violence of cutting.

**Test:** Can the user see that every clip was taken from somewhere?

---

# 13. Used/Unused Source Memory Heuristic

**Rule:** The source strip should remember what the timeline has taken.

Source slices should show:

| Slice state | Meaning                     |
| ----------- | --------------------------- |
| untouched   | available                   |
| hovered     | calling                     |
| dragged     | being extracted             |
| used        | already placed              |
| active      | currently under playhead    |
| ghost-used  | used, but source is missing |

**Design implications:**

* The source is not just input.
* It becomes an archive of editorial attention.
* The user can see both selection and omission.

**Avoid:** source strips that reset visually after a drag.

**Test:** Can the user ask, “What have I not used yet?” and answer visually?

---

# 14. Live Monitor as Surveillance Heuristic

**Rule:** The monitor is not merely confirmation. It watches the edit become real.

The monitor should label what kind of reality it is showing:

* `SOURCE PREVIEW`
* `SEQUENCE OUTPUT`
* `EMPTY TIME`
* `GHOST OUTPUT`
* `PROXY APPARITION`
* `FAILED BODY / ARMED SOURCE`

**Design implications:**

* Use monitor overlays for state clarity.
* Let the monitor display proxies for missing media.
* Do not leave ambiguous black screens.

**Avoid:** unlabeled preview states.

**Test:** Does the user always know whether they are seeing source, sequence, gap, or ghost?

---

# 15. Performance Console Heuristic

**Rule:** The editor should be playable.

The interface should support fast, embodied, repeatable gestures.

**Design implications:**

* Keyboard shortcuts should feel like instrument keys.
* Number keys select sources.
* Space invokes playback.
* Dragging slices should be low-friction.
* Rolling reels should alter source access rhythm.
* EDL export should feel like capturing a performance trace.

**Avoid:** heavy dialog flows and modal panels.

**Test:** Can someone use the tool rhythmically, almost musically?

---

# 16. Archive of Absences Heuristic

**Rule:** A saved project should remember both what appeared and what failed to appear.

The project file should preserve:

* loaded source names
* missing source names
* ghost status
* failed decode states
* relink history
* sequence structure
* empty gaps
* source-roll offsets
* placed slices

**Design implications:**

* Saving should not require perfect media.
* Loading should show absences as meaningful.
* Relinking should be visible as a historical act.

**Avoid:** reducing project state to only currently valid media.

**Test:** Does the project remain meaningful even when no original videos are available?

---

# 17. Montage Oracle Heuristic

**Rule:** The system may suggest collisions, but it should not over-explain them.

If generative or random functions are added, they should behave like an oracle, not an assistant.

Possible actions:

* suggest strange slice pairings
* shuffle unused fragments
* place ghost clips
* generate a ritual grid pattern
* propose a cut based on contrast

**Design implications:**

* Keep suggestions optional.
* Let the user accept, mutate, or ignore.
* Do not explain too much.
* Preserve mystery.

**Avoid:** productivity-style AI that says, “Here is the optimal edit.”

**Test:** Does the suggestion provoke thought rather than automate judgment?

---

# 18. Negative-Space Feature Heuristic

**Rule:** Every new feature must protect the weird core.

Before adding anything, ask:

> Does this make the system more like a normal editor, or more like itself?

If it makes the tool more generic, reject or constrain it.

## Addition/subtraction examples

| Proposed addition | Required subtraction                                        |
| ----------------- | ----------------------------------------------------------- |
| More sources      | Collapse inactive source visuals                            |
| Render export     | Reduce EDL prominence or create separate ritual export mode |
| Audio             | Remove another persistent visual layer                      |
| Effects           | Limit them to ghost/proxy states only                       |
| AI suggestions    | Remove generic editing commands                             |
| Timeline zoom     | Hide it until repeated trim friction appears                |
| Media bin         | Remove A/B/C/D ritual source model — probably not worth it  |

**Test:** Does the addition increase clarity, or merely increase capability?

---

# 19. Anti-NLE Heuristic

**Rule:** Never compete with professional video editors on their terms.

This system should win by being stranger, faster, more tactile, and more conceptually alive.

Do not optimize for:

* complete post-production
* media management
* effects pipelines
* precision color/audio work
* delivery formats
* client review workflows

Optimize for:

* source awakening
* fragment extraction
* rapid montage
* visible omission
* ghost composition
* performable sequence logic
* exportable temporal traces

**Test:** Would this feature make the tool easier to compare to Premiere? If yes, be suspicious.

---

# 20. Master Heuristic

> Treat every limitation as a possible instrument.

No audio becomes visual percussion.
No render becomes live performance.
Decode failure becomes ghost media.
Fixed grid becomes ritual meter.
Relinking becomes reinterpretation.
JSON becomes a spellbook.
One-second clips become seeds.
The rail becomes an altar.

The control surface should not ask:

> How do we make this a better video editor?

It should ask:

> How do we make visible splicing feel inevitable, strange, and alive?
